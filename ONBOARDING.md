## 1. Обзор проекта

**smit_billing** — Django-система биллинга и управления абонентами для интернет-провайдера.

| | |
|---|---|
| **Язык** | Python |
| **Фреймворки** | Django, Celery, Django REST Framework, Docker |
| **Архитектура** | Монолит Django с тремя «фасадами»: операторский веб-интерфейс, личный кабинет клиента, мобильный REST API |
| **Точка входа** | `manage.py` → `config/settings.py` → `config/urls.py` / `config/celery.py` |

Система решает полный цикл провайдера: учёт абонентов и тарифов, начисление абонплаты,
приём платежей (ЮКасса, W1, банковские выписки), управление сетевым доступом через RADIUS/NAS,
фискализацию по 54-ФЗ, выгрузки СОРМ и асинхронную обработку через Celery.

## 3. Ключевые концепции

- **Три фасада, одно ядро.** `billing/views` (оператор), `lk` (клиент) и `mobile_api` (мобайл)
  — три разных интерфейса поверх одних и тех же моделей `billing/models`. Изучив модели, вы
  понимаете все три.
- **Собственный UI-фреймворк операторки.** Почти каждая страница оператора наследуется от
  базовых классов в `billing/views/form.py` (`BaseView`, `SearchFormMixin`, `ReportMixin`,
  `FormWizard`). Это самый переиспользуемый файл интерфейса — поймите его первым.
- **Абонент — центр графа.** Модель абонента (`billing/models/abonents.py`) хранит IP, тариф,
  баланс и связывает практически все подсистемы.
- **`billing/urls.py` — узел №1 по связности** (73 связи): диспетчер, соединяющий запросы оператора
  с представлениями. Хорошая отправная точка для навигации «откуда что вызывается».
- **Интеграция с legacy EBS.** Много management-команд и сервисов (`sync_from_ebs`, `ebs_client`)
  — это миграция/синхронизация со старой биллинговой системой EBS.
- **Регуляторика встроена в код.** Фискализация (54-ФЗ, АТОЛ) и СОРМ — отдельные подсистемы
  с собственными моделями, сервисами и Celery-задачами.

## 4. Гид по системе (рекомендуемый путь изучения)

Порядок выстроен по зависимостям: от фундамента к фасадам и интеграциям.

1. **Точка входа и конфигурация** — `manage.py`, `config/settings.py`, `config/urls.py`, `config/celery.py`.
   Фундамент: как стартует система и где собираются маршруты трёх приложений.
2. **Ядро моделей: абонент, тариф, услуга** — `billing/models/{users,abonents,tariff,usluga}.py`.
   Три сущности, связывающие всё остальное.
3. **Справочники и инфраструктурные модели** — `billing/models/{lookups,nas,system_settings}.py`.
   Словари (категории, NAS, IP-пулы, VLAN, адресное дерево), к которым обращается вся система.
4. **UI-фреймворк оператора** — `billing/views/form.py` + `billing/urls.py` + `billing/auth_backend.py`.
   Как устроена любая страница операторки.
5. **Карточка абонента и финансовые операции** — `billing/views/{users,users_ajax,finance_operations,finops_list}.py`.
   Главный объект ежедневной работы: блокировка, смена тарифа, пополнение, журнал операций.
6. **RADIUS и сетевой доступ** — `billing/views/nas.py`, `billing/services/nas_control.py`, `billing/tasks/nas_sync.py`, `billing/views/pools.py`.
   Авторизация абонентов, мониторинг SLA, выделение IP, принудительное отключение.
7. **Личный кабинет (ЛК)** — `lk/views/{auth,dashboard,payments,promise_pay,chat}.py`, `lk/decorators.py`.
   Самообслуживание абонента + AI-чат поддержки.
8. **Мобильный API (DRF)** — `mobile_api/{authentication,serializers,urls}.py`, `mobile_api/views/`.
   REST для iOS/Android, JWT, throttling.
9. **Платежи и банковские выписки** — `billing/services/bank_parsers/`, `bank_matching.py`, `billing/views/payment_settings.py`.
   Парсинг выписок Альфа/Сбер и сопоставление платежей с абонентами.
10. **Фискализация (АТОЛ / ОФД)** — `billing/fiscal/{manager,providers/atol,providers/base}.py`, `billing/tasks/fiscal.py`.
    Отправка чеков по 54-ФЗ.
11. **СОРМ** — `billing/models/sorm.py`, `billing/services/sorm_sql.py`, `billing/tasks/sorm_export.py`.
    Выгрузки для регулятора (Приказ 573).
12. **Celery-задачи и операционка** — `billing/tasks/{billing_worker,abonent_block,tarif_change,messaging}.py`, `management/commands/billing_preflight.py`.
    Основной биллинговый воркер, автоблокировка должников, рассылки, диагностика.

## 6. С чего начать новому разработчику

1. Прочтите §1–§3 этого документа и шаги 1–4 гида — это ~15 файлов, дающих 80% понимания.
2. Откройте интерактивный дашборд (`/understand-dashboard`) и пройдите тур визуально —
   узлы подсвечены по слоям, видно «кто кого импортирует».
3. Для конкретной задачи найдите нужный слой в таблице §2 и идите от `*/urls.py` соответствующего
   приложения к нужному view.
