HelpDesk — система поддержки клиентов

HelpDesk — рабочее место операторов техподдержки, развёрнутое на open-source-движке FreeScout. Установлен на поддомене support.smit34.ru и двусторонне интегрирован с СмИТ Биллингом: данные абонента подгружаются в карточку тикета, тикеты доступны во вкладке «HelpDesk» абонента, AI-ассистент Claude отвечает на простые вопросы автоматически.

1. Обзор системы

HelpDesk закрывает четыре задачи техподдержки:

  • Регистрация обращений — клиент пишет на email support@, в ЛК, в мобильном приложении, через VK или Telegram-бот — обращение автоматически превращается в тикет;
  • Распределение работы — тикеты раскладываются по ящикам (Поддержка, Подключение, Разработка), назначаются оператору, попадают на канбан-доску;
  • Ответ клиенту — оператор видит баланс, тариф, адрес абонента прямо в окне тикета (виджет SMIT Billing), отвечает в один клик;
  • Аналитика и история — отчёты по SLA, продуктивности, отзывам; полный архив переписки с клиентом.
Технологическая база. Движок — FreeScout 1.8.221 (PHP/Laravel). Поверх него стоят наши модули: SmitUI (тёмная тема + брендинг), SmitBilling (виджет данных абонента в боковой панели), ClaudeAI (AI-ассистент), KnowledgeBase (база знаний), Kanban, AmoCRMSync, VK/Telegram/VoiceMail-интеграции. Хостинг — отдельный сервер support.smit34.ru, БД — MySQL helpdesk. Связь с биллингом — через REST API https://rbill.smit34.ru/rest_api/v2/.

Ящики и роли

В HelpDesk заведено три рабочих ящика, каждый со своими операторами и SLA:

ЯщикEmailНазначение
Поддержка клиентовask@support.smit34.ruВсе обращения по сервису: интернет не работает, плохая скорость, оплата, услуги, переподключение
Подключениеconnect@support.smit34.ruЗаявки на новое подключение — попадают на отдельную канбан-доску (Новый лид → Тариф выбран → Назначен монтаж → Подключено)
Разработкаdevelopment@support.smit34.ruВнутренний канал: обращения от операторов к разработчикам биллинга (баги, доработки)

Помимо них есть служебные ящики (Звонки, Поддержка ROBOR) — они скрыты от обычных операторов через модуль SmitUI и используются только для технических автоматизаций (например, расшифровка голосовых сообщений с АТС попадает в «Звонки» как тикет с тегом канала).

2. Рабочий процесс

Стандартный цикл от входящего обращения до закрытия тикета:

  1. Поступление — клиент написал в любой из каналов (email, ЛК, мобильное приложение, VK-сообщество, Telegram-бот). HelpDesk автоматически создаёт тикет и кладёт его в нужный ящик в статус «Неназначен»;
  2. Обогащение — модуль SmitBilling ищет абонента по email или телефону через REST API биллинга. Если абонент найден — в правой панели тикета показывается баланс, тариф, адрес, кнопки «Открыть в биллинге», «Создать платёж», «Перезагрузить роутер»;
  3. Первый ответ AI (опционально) — если включён ClaudeAI и вопрос распознан как простой (баланс, как пополнить, забыл пароль), Claude отвечает автоматически в течение нескольких секунд. Если вопрос требует человека — Claude отвечает «передаю оператору» и помечает тикет тегом escalated;
  4. Назначение оператору — старший смены или сам оператор берёт тикет в работу (статус «В работе», назначен ответственный, тикет уходит из общего списка «Неназначен» в «Мои»);
  5. Диалог — оператор и абонент переписываются. Каждый ответ оператора уходит абоненту тем же каналом, что и пришло обращение (email → email, VK → VK-сообщение, мобильное приложение → push + отображение в чате ЛК);
  6. Закрытие — тикет переводится в статус «Решён». Клиенту приходит письмо с ссылкой на оценку качества (1–5 звёзд) — оценки попадают в отчёт «Удовлетворённость».
Важная связь с биллингом: событие закрытия тикета (conversation.status_changed(closed)) отправляется webhook'ом в биллинг и записывается в журнал AUDIT_OPERATIONS абонента — для дальнейших отчётов «сколько обращений у должников», «топ-проблем за месяц».

3. Панель управления

Главный экран HelpDesk (support.smit34.ru) — обзор по всем ящикам сразу. По каждому ящику показано количество неназначенных, моих, отмеченных, черновиков, назначенных тикетов и дата последнего обращения.

  • Цифра в кружке рядом с «Неназначены» — количество тикетов в очереди (если ≥1 — нужно разобрать);
  • Кнопка-шестерёнка — настройки ящика (доступ операторов, шаблоны автоответов, теги);
  • Стрелка — открыть список тикетов ящика.

На мобильном устройстве (планшет/смартфон) панель адаптируется в одну колонку — карточки ящиков идут одна под другой, в шапке остаётся только бургер-меню, поиск и тема. Подходит для оператора, который начинает смену вне офиса.

4. Списки тикетов

Внутри ящика (например «Поддержка клиентов») — таблица тикетов. По умолчанию открыта папка «Неназначены», слева — переключатель папок (Все, Мои, Неназначены, Отмеченные, Черновики, Закрытые, Спам), справа — фильтры по статусу и тегам.

Каждая строка таблицы — это:

  • Имя клиента и email/телефон (телефонные тикеты — из голосовой почты или из мобильного приложения);
  • Цветные бейджи:
    • смит / другие теги — категория проблемы (см. раздел 7);
    • −123₽ — отрицательный баланс абонента (модуль SmitBilling подтягивает в реальном времени);
    • 450₽ — положительный баланс;
  • Превью первой строки сообщения — для быстрого понимания сути обращения без открытия;
  • Время — когда пришло обращение (или последний ответ).

Над таблицей — строка массовых действий: отметить, переместить в другой ящик, назначить оператора, изменить статус, удалить. Действия применяются к отмеченным тикетам (checkbox в левой колонке).

5. Карточка тикета

Клик по тикету открывает карточку — основной экран оператора. Слева — переписка, справа — данные абонента, в верхней части — панель быстрых действий.

Верхняя панель действий

  • Назад — к списку ящика;
  • Редактировать — изменить тему, заказчика, теги;
  • Удалить — переместить в корзину (восстанавливается 30 дней);
  • Отложить — спрятать тикет до указанной даты (например «вернуться через 3 дня после обещанного платежа»);
  • Теги — навесить категории;
  • Ещё — пересылка на другой ящик, объединение с другим тикетом, печать, экспорт.

Справа в шапке — кнопки «Активен / Отвечен / Закрыт / Спам» (статус) и «Любой / Иванов / Петров» (ответственный оператор).

Правая панель — данные абонента (модуль SmitBilling)

Если абонент найден в биллинге по email или телефону, в правой колонке появляется блок:

  • ФИО абонента с ссылкой на полную карточку в админке биллинга;
  • Контакты — email, телефон (рабочий/мобильный/voicemail);
  • Договор и тариф — номер договора, действующий тариф, статус (активен / заблокирован);
  • Баланс — цветным бейджем (красный отрицательный / зелёный положительный) с возможностью клика для создания финоперации;
  • Кнопки быстрых действий:
    • Claude AI — подсказка ответа от AI на основе истории абонента (тариф, баланс, последние обращения);
    • Черновик — сохранить заготовку ответа без отправки;
    • Кнопки оборудования (Пинг роутера, Перезагрузка ONT и т.д.) — если NAS поддерживает удалённое управление.
Если абонент не найден — вместо данных биллинга показывается «Клиент не найден в биллинге». Оператор может вручную сопоставить тикет с абонентом, нажав «Связать» и выбрав договор из выпадающего списка с поиском.

Окно ответа

Под перепиской — редактор ответа: форматирование (жирный/курсив/списки/вставка ссылки/изображения), кнопка вставки шаблона, выбор статуса после отправки, назначение оператора. Кнопки внизу:

  • AI Подсказка — Claude предложит черновик ответа на основе истории тикета и данных абонента;
  • Отправить ответ — отправляется клиенту тем же каналом, что пришло обращение;
  • Переслать — отправить тикет с историей другому оператору или внешнему адресату;
  • Добавить внутреннее примечание — комментарий, видимый только операторам (не уходит клиенту);
  • Создать — создать отдельный тикет из текущего сообщения (например выделить новую проблему).

6. Канбан-доска подключений

Для ящика «Подключение» включён модуль Kanban — заявки на подключение отображаются как карточки на доске со столбцами по этапам сделки.

Столбцы доски

  1. Новый лид — поступило обращение, абонент пока не подтвердил серьёзность намерений;
  2. Тариф выбран — менеджер согласовал с клиентом тариф и адрес подключения;
  3. Назначен монтаж — на адрес назначена бригада монтажников и дата;
  4. Подключено — клиент работает, договор оформлен (в биллинге создаётся абонент);
  5. Отказ / Нет покрытия — заявка закрыта без подключения (с указанием причины для аналитики).

Что в карточке

  • Имя контактного лица и адрес;
  • Канал источника (AI/чат-бот, форма с сайта, телефон, email) — цветная плашка;
  • Теги (см. раздел 7);
  • Заметки менеджера (тариф, особенности подключения, сроки);
  • Номер тикета и время создания;
  • Назначенный менеджер.

Карточки перетаскиваются между столбцами мышью — статус тикета меняется автоматически. Внизу доски — кнопка «Показать закрытые (N)» для просмотра завершённых сделок.

7. Теги и фильтры

Теги — главный инструмент классификации тикетов. По тегам ведётся аналитика (топ-проблем за месяц), фильтрация в списках, автоматизация (например, тег оплата + неотвеченный ≥ 4 часа → SMS-напоминание оператору).

Категории тегов в нашей инсталляции

ТегНазначение
смит, роборКакой бренд оператора затронут (если у нас несколько ISP)
нет-интернета, los, скорость, роутер, проблемная-зонаТип технической проблемы
подключение, повторное, оплатаТип запроса
активен, заблокированТекущий статус абонента (проставляется автоматически по данным биллинга)
бизнесКорпоративный клиент (приоритет)
vk, voiceКанал источника
Номер телефона (например 78442554065)Тикеты, пришедшие с конкретного номера АТС

Рядом с каждым тегом — счётчик активных тикетов. Клик по тегу открывает список всех тикетов с этим тегом по всем ящикам — удобно для проверки «сколько сейчас аварийных обращений «нет-интернета».

Автотеги. Часть тегов проставляется автоматически: vk и voice — по источнику обращения, заблокирован / активен — модулем SmitBilling по данным биллинга, теги номеров — по расшифровке голосового сообщения с АТС.

Фильтрация в списках

Над таблицей тикетов в любом ящике — фильтр «Теги». Можно выбрать несколько тегов и одну из двух стратегий: «любой из» (OR) или «все одновременно» (AND). Фильтр сохраняется в URL — можно скопировать ссылку и отправить коллеге.

8. База знаний

База знаний — структурированный каталог типовых сценариев решения проблем. Используется операторами при обучении, ботом AIDA для авто-ответов, и Claude AI как контекст для подсказок.

Структура базы

База разделена на четыре вкладки в верхней панели:

  • Сценарии решения — пошаговые инструкции (S01 «Нет интернета», S02 «Горит LOS», S03 «Проблемы с MK», S04 «Низкая скорость», S05 «Проблемы с роутером», S06 «Проблемы с Wi-Fi» и т.д.). У каждого сценария — приоритет (высокий/средний/низкий) и аудитория (Бот / Оператор / Бот+Оператор);
  • Дерево диагностики — иерархия вопросов «если LOS красный → проверить муфту, если зелёный → проверить настройки роутера»;
  • Шаблоны ответов — заготовки текстов для типовых ответов (форматирование, переменные {ФИО}, {баланс}, {тариф}, подставляются из данных биллинга);
  • Аналитика — отчёты по обращениям: топ-проблем за период, эффективность сценариев, скорость решения по типу.

В шапке базы есть кнопка «Работа с сервисом ↗» — ведёт на эту страницу документации (раздел про FreeScout) в новой вкладке.

Аудитория «Бот»

Сценарии с пометкой «Бот» используются AI-ассистентом Claude для автоматических ответов в чате ЛК и мобильного приложения. Если сценарий простой («забыл пароль», «как пополнить»), Claude отвечает по сценарию сам. Если требуется участие оператора («медленный интернет, проверяли модем») — Claude отвечает фразой «передаю оператору» и создаёт тикет в FreeScout с тегом сценария.

Аудитория «Оператор»

Сценарии для оператора — это шпаргалка в карточке тикета. По клику на «AI Подсказка» Claude находит подходящий сценарий по тексту обращения и предлагает черновик ответа на основе сценария + данных абонента.

9. AI-ассистент Claude

Модуль ClaudeAI подключает AI-ассистент Anthropic к FreeScout. Используется в трёх режимах:

  1. Автоответ в чате ЛК и мобильного приложения — абонент пишет в чат, ClaudeAI отвечает сам, если вопрос соответствует базе знаний с аудиторией «Бот». При непонятной ситуации — фраза «передаю оператору» и создание тикета в FreeScout;
  2. Подсказка ответа оператору — в карточке тикета кнопка «AI Подсказка» (или «Claude AI» в правой панели). Claude получает контекст: текст тикета, данные абонента (баланс, тариф, история обращений), подходящие сценарии из базы знаний — и формирует черновик ответа. Оператор редактирует и отправляет;
  3. Классификация и теги — Claude автоматически проставляет теги по содержанию тикета (например, «нет-интернета» + «робор», если в тексте упомянуты обе характеристики).

Настройка

В админке HelpDesk: support.smit34.ru/claudeai/settings (только администратор). Параметры:

  • API-ключ Anthropic — выдаётся в личном кабинете Anthropic Console;
  • Прокси — API api.anthropic.com заблокирован из российских IP, поэтому запросы идут через прокси (настройки из биллинга, раздел Telegram-прокси переиспользуется);
  • URL биллинга — https://rbill.smit34.ru/rest_api/v2/ — откуда Claude берёт контекст по абоненту;
  • Токен биллинга — Bearer-токен для аутентификации в REST API биллинга.
Резервный AI — AIDA. Если Claude недоступен (отсутствует ключ, не работает прокси, превышен лимит API), используется собственная модель AIDA — fallback для критичных автоответов. Настройки: раздел «AIDA» в админке биллинга.

Расходы и лимиты

Claude API оплачивается за токены (входные + выходные). Один типовой ответ — около 500–2000 токенов. В админке биллинга есть отчёт «AI-сессии» (раздел /ai-sessions в HelpDesk) — статистика по дням, типам сессий (текст/голос), провайдерам, ошибкам. Используется для контроля расходов.

10. Связь с биллингом

HelpDesk и СмИТ Биллинг связаны двусторонне. Подробное техническое описание интеграции — в разделе «Настройки биллинга → FreeScout-интеграция». Здесь — краткая сводка по точкам соединения.

11. Администрирование

Управление операторами

Настройки → Агенты. Создание нового оператора, выдача доступа к ящикам, выбор роли (Оператор / Администратор), смена пароля, деактивация. Деактивированные операторы остаются в списке для истории (их подписи на тикетах сохраняются), но не могут войти.

Ящики и каналы

Настройки → Ящики. Для каждого ящика настраивается:

  • Входящая почта (IMAP) — Mailcow на mail.support.smit34.ru;
  • Исходящая почта (SMTP) — тот же Mailcow;
  • Автоответ — шаблон письма-подтверждения «ваш тикет № принят»;
  • Доступ операторов — кто видит этот ящик;
  • Шаблонные ответы — общие для ящика;
  • Пользовательские поля Customer — billing_id, contract_number, tariff, balance (заполняются автосинхронизацией с биллингом);
  • Канбан-доска (для ящика «Подключение») — настройка столбцов и автоматизаций.

Интеграции каналов

В разделах Настройки → VK Integration / TelegramNotification / VoiceMail настраиваются:

  • VK — ID сообщества, токен, обработка входящих сообщений как тикетов;
  • Telegram — bot token, чаты для алертов (распределение по топикам по типам событий);
  • VoiceMail — обработка email от АТС с расшифровкой голоса и автоматическое создание тикета с привязкой к абоненту по номеру.

Отчёты

Раздел Отчёты в верхней панели:

  • По тикетам — динамика поступления, среднее время первого ответа, среднее время решения, по тегам/ящикам/операторам;
  • По продуктивности — кто сколько отвечает, среднее время на тикет;
  • По отзывам — оценки клиентов (1–5 звёзд), средний балл, рекомендации;
  • По затраченному времени — если включён модуль учёта рабочего времени (Time Tracking).

Резервное копирование

HelpDesk развёрнут на отдельном сервере и бэкапится отдельно от биллинга:

  • MySQL helpdesk — ежедневный mysqldump;
  • Файлы вложений — /var/www/freescout/storage/app/attachments/ — еженедельный tar.gz;
  • Кастомные модули (SmitUI, SmitBilling, ClaudeAI, KnowledgeBase и др.) — в git-репозитории ircitdev/SmitBilling и связанных.

Обновление FreeScout

HelpDesk обновляется через встроенный апдейтер (Настройки → Статус → Доступна версия...). Процедура:

  1. Остановить очередь и cron: systemctl stop freescout-queue.service;
  2. Сделать резервную копию storage/ и БД;
  3. Нажать «Обновить» в админке;
  4. После обновления — проверить, что апдейтер не затёр кастомные blade-шаблоны (логин, восстановление пароля, оформление email) — если затёр, восстановить из бэкапа;
  5. Очистить кеш: php artisan config:clear && cache:clear && view:clear;
  6. Запустить очередь: systemctl start freescout-queue.service.
Запасной канал на случай недоступности HelpDesk: в админке биллинга есть раздел «Старые HDSK» — это архив старого встроенного хелпдеска СмИТ Биллинга, который использовался до апреля 2026. На него можно временно переключить операторов, если FreeScout полностью недоступен. Для перехода назад в FreeScout — все новые тикеты надо будет вручную мигрировать.
